0x00 背景

LLVM作者是Chris Lattner，绝对的大神，这位大神在硕博期间主要研究方向就是编译器优化相关的课题，当然就这一方向发表论文无数，在读博期间提出一套在编译时、链接时、运行时以及闲置时的优化策略、从此LLVM基本思想被确立，奠定了其在编译器圈的地位。 同时作为老牌编译器gcc,苹果一直在用，但是GCC对Objective-C的语言特性支持不够，苹果又无法敦促其快速响应，自己开发的代码模块又不能得到gcc委员会的合并，惹怒乔公，所以待到Chris Lattner毕业，就将其纳入苹果麾下，安心开发自己的编译器，LLVM也事实上成为了苹果官方编译器。

LLVM（Low Level Virtual Machine）全称底层虚拟机，当初这哥们设计之时是为了做优化方面的研究，所以搭建了一套虚拟机进行代码转换，这不现在其实是编译器了，于是官方放弃了原来的全称称呼，但是简称LLVM被保留并延续下来，现在是项目名称不是缩略词。

LLVM期初是编译器框架，所以需要一个前端来支撑整个系统，所以Apple组织人手开发了Clang，Clang可以编译C、C++和Objective-C，后来Apple推出了Swift，也是基于LLVM编译器框架来支持的，当然Kotlin，Ruby，Python，java等等语言都可以用LLVM编译。 Chris Lattner这位大神，17年跳到特斯拉，再后来去了Google，在TensorFlow团队，确实神。后来LLVM开源之后由LLVM委员会负责维护，发展势头很猛。

0x01 编译器设计

LLVM受到世人关注是因为它有独立完善严格约束的中间代码表示。中间代码可以认为是LLVM的字节码，LLVM抽象的精髓，前端生成中间代码，后端自动进行各类优化分析，理论上可以支持任何开发语言。

传统编译器架构设计：

  • 前端 Frontend
    编译器前端的任务是解析源代码（编译阶段），进行词法分析、语法分析、语义分析，然后构建抽象语法树（Abstract Syntax Tree AST）。

  • 优化器 Optimizer
    在前段基础上优化器负责各种中间代码优化，改善代码的运行时间。

  • 后端 Backend（代码生成器 Code Generator）
    将代码映射到目标指令集，生成各自平台的机器代码。 传统编译器代表gcc。

LLVM架构设计：

  • 不同前端后端有了统一的桥梁–中间代码LLVM Intermediate Representation (LLVM IR)
  • 如果需要支持一种新的编程语言，那么只需要实现一个新的前端。
  • 如果需要支持一种新的硬件设备，那么只需要实现一个新的后端。
  • 优化阶段是一个通用的阶段，因为统一了LLVM IR，不论是新的编程语言，还是新的硬件设备，都不需要对优化阶段做修改。
  • 相比之下，GCC的前端后端耦合在一起，所以GCC要支持一门新的语言，或者支持一个新的目标平台，改起来就会困难。
  • LLVM如今已作为实现各种静态和运行时编译语言的通用基础结构(GCC家族、Java、.NET、Python、Ruby、Kotlin、Haskell、D等等)

0x02 Clang(官方标准读音[klæŋ])简介

LLVM项目的一个子项目，基于LLVM架构的C/C++/Objective-C编译器前端。 特点：

  • 编译速度快:在某些平台上，Clang的编译速度显著的快过GCC。
  • 占用内存小:Clang生成的AST所占用的内存比GCC小。
  • 模块化设计:Clang采用基于库的模块化设计，易于 IDE 集成及其他用途的重用。
  • 诊断信息可读性强:在编译过程中，Clang创建并保留了大量详细的元数据 (metadata)，有利于调试和错误报告。
  • 设计清晰简单，容易理解，易于扩展增强。

语法树-AST(Abstract Syntax Tree)

抽象语法树能清晰展现程序的脉络，方便编译过程的后续环节（如代码生成器）对程序进行解读。AST是开发者为语言量身定制的一套模型，基本上开发语言中的结构都有一种AST对象来对应。

/// ExprAST - Base class for all expression nodes.
class ExprAST {
public:
  virtual ~ExprAST() = default;

  virtual Value *codegen() = 0;
};

/// NumberExprAST - Expression class for numeric literals like "1.0".
class NumberExprAST : public ExprAST {
  double Val;

public:
  NumberExprAST(double Val) : Val(Val) {}

  Value *codegen() override;
};

定义了基类ExprAST，其中子类NumberExprAST将数值常量的值存放在成员变量中，以备编译器后续查询，用于表示数值常量的子类。其他各种表达式的AST都可以通过继承来实现。

0x03 应用

要实现一门新的开发语言，首先要能处理文本文件（源码），古法传统会先利用“词法解析器”将输入切成“标记（token）”，词法解析器返回的每个语元都带有一个token编号，所以要把新语言中要出现的token都作定义。 再搞定语法解析器之后，就要定义并构造抽象语法树了， 我们可以根据基类ExprAST，来设计几个必要的类，做些简单的事，比如定义VariableExprAST用于保存变量名，BinaryExprAST用于保存运算符（如“+”），CallExprAST保存函数名和参数列表等。

/// VariableExprAST - Expression class for referencing a variable, like "a".
class VariableExprAST : public ExprAST {
  std::string Name;

public:
  VariableExprAST(const std::string &Name) : Name(Name) {}

  Value *codegen() override;
  const std::string &getName() const { return Name; }
};

/// BinaryExprAST - Expression class for a binary operator.
class BinaryExprAST : public ExprAST {
  char Op;
  std::unique_ptr<ExprAST> LHS, RHS;

public:
  BinaryExprAST(char Op, std::unique_ptr<ExprAST> LHS,
                std::unique_ptr<ExprAST> RHS)
      : Op(Op), LHS(std::move(LHS)), RHS(std::move(RHS)) {}

  Value *codegen() override;
};
/// CallExprAST - Expression class for function calls.
class CallExprAST : public ExprAST {
  std::string Callee;
  std::vector<std::unique_ptr<ExprAST>> Args;

public:
  CallExprAST(const std::string &Callee,
              std::vector<std::unique_ptr<ExprAST>> Args)
      : Callee(Callee), Args(std::move(Args)) {}

  Value *codegen() override;
};

在构造AST之前，还需要先准备用于构造AST的语法解析器，利用解析器。

/// identifierexpr
///   ::= identifier
///   ::= identifier '(' expression* ')'
static std::unique_ptr<ExprAST> ParseIdentifierExpr() {
  std::string IdName = IdentifierStr;

  getNextToken(); // eat identifier.

  if (CurTok != '(') // Simple variable ref.
    return std::make_unique<VariableExprAST>(IdName);

  // Call.
  getNextToken(); // eat (
  std::vector<std::unique_ptr<ExprAST>> Args;
  if (CurTok != ')') {
    while (true) {
      if (auto Arg = ParseExpression())
        Args.push_back(std::move(Arg));
      else
        return nullptr;

      if (CurTok == ')')
        break;

      if (CurTok != ',')
        return LogError("Expected ')' or ',' in argument list");
      getNextToken();
    }
  }

  // Eat the ')'.
  getNextToken();

  return std::make_unique<CallExprAST>(IdName, std::move(Args));
}

通过以上函数就可以构造各种AST节点，解析各种表达需要添加一个统一入口

/// primary
///   ::= identifierexpr
///   ::= numberexpr
///   ::= parenexpr
///   ::= ifexpr
///   ::= forexpr
///   ::= varexpr
static std::unique_ptr<ExprAST> ParsePrimary() {
  switch (CurTok) {
  default:
    return LogError("unknown token when expecting an expression");
  case tok_identifier:
    return ParseIdentifierExpr();
  case tok_number:
    return ParseNumberExpr();
  case '(':
    return ParseParenExpr();
  case tok_if:
    return ParseIfExpr();
  case tok_for:
    return ParseForExpr();
  case tok_var:
    return ParseVarExpr();
  }
}

就按照这样的套路自定义语法，通过llvm框架可以很快实现一个编译器支持一门新语言，你可以开发Xlang、Ylang、Zlang都没有问题，后面就是相关生态和语言本身的设计哲学了。 当然后面还需要做IR生成，JIT与优化，实现语言的流程控制，运算符等等一些列完善后就造出一门新的开发语言了。
